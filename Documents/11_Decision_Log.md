# BusinessOS AI — Project Context & Decision Log

*Purpose: preserve the reasoning behind architectural decisions, not just the decisions themselves, so the project doesn't have to be "re-explained" to a new session or teammate.*

| # | Decision | Why |
|---|---|---|
| 1 | Product is a cross-department platform, not a single-department tool | Original problem statement: departments run on disconnected software; the goal is unified visibility, not another single-purpose tool. |
| 2 | One Planner Agent with two entry points, instead of two separate reactive/proactive architectures | Simpler to build and reason about; mirrors how real systems (e.g. Gmail: user-composed vs. received mail) share one core engine regardless of trigger source. Rejected the two-diagram version because it implied two systems, which isn't true — the middle of the pipeline is identical either way. |
| 3 | Renamed "Business Sentinel Agent" → "Enterprise Intelligence Engine" | The original name made it sound like just another specialized agent. Renaming it signals it's an architectural layer with a distinct responsibility (continuous monitoring), which is also the project's USP. |
| 4 | Role-based access enforced at the MCP/data-access layer, not the frontend | A professor flagged basic RBAC as "not a differentiator" when described as a UI feature. Moving enforcement to the data-access layer makes it a real security architecture decision, not a cosmetic one, and it's also just correct practice regardless of grading. |
| 5 | Agents hold no persistent business data themselves | Keeps agents stateless and simple; all real data always comes fresh from MCP, avoiding sync/staleness issues between an agent's internal state and the source of truth. |
| 6 | Alerts are filtered by role + severity before delivery | Directly prevents "alert fatigue" (identified as a risk) — the proactive mode is only valuable if it's trustworthy signal, not noise. |
| 7 | Simulated CRM/ERP/HRMS via public datasets (e.g. Kaggle) for the workshop build | No real company data available; simulated data lets the architecture be fully demonstrated without needing real integrations, which are explicitly out of scope for the 5-day MVP. |
| 8 | Explainability/evidence trail is a required field on every answer/alert, not optional | Chosen as part of the USP brainstorm (Explainable AI) — addresses a real, common failure mode of AI adoption (users don't trust black-box outputs). |
| 9 | 5-day build plan sequences DB → MCP/RBAC → Planner → agents (Sales, Finance, HR) → Sentinel/Alerts → frontend | Backend-first, and specifically access-control-first, so RBAC isn't retrofitted at the end; frontend is scaffolded last since it carries the least unique logic. |
| 10 | Removed the entire legacy scaffold (`app/agents/{base_agent,planner,sales_agent,finance_agent,hr_agent}.py`, `app/mcp/*`, `app/sentinel/*`, `app/alerts/*`, `app/api/{routes_ask,routes_alerts,routes_admin,routes_auth}.py`, `app/db/*`) instead of finishing or reconciling it | An architecture review found two incompatible, same-named implementations coexisting (`AgentResult`, `BaseAgent`, `PlannerAgent`/`SalesAgent`/`FinanceAgent`) -- one matching this log's original spec literally, one actually wired into `demo.py`/FastAPI. Grepped the whole repo first to confirm zero live code imported the legacy cluster before deleting. The "seams" it partially represented (Data Access Layer, Sentinel, Alerts, DB) are rebuilt properly in decisions #12-#13 and deferred (not deleted-and-forgotten) per decision #16. |
| 11 | Kept the as-built per-domain folder layout (`/sales`, `/finance`, `/enterprise`, `/ai`, `/planner`, each with its own `agent.py`/`analytics_service.py`/`data_access.py`/`query_mapper.py`) instead of reverting to CLAUDE.md's original `/agents`-centric structure | The per-domain layout already had full test coverage and is more consistent (one place to look for "everything about Sales") than the original `/agents` flat folder. Per explicit instruction, updated `CLAUDE.md` to describe the as-built structure as the source of truth rather than rewriting working, tested code to match a doc that predated the Enterprise/AI Agents. `Documents/04_System_Architecture.md` and `05_Agent_Design.md` got a status note pointing back to `CLAUDE.md` rather than a full rewrite, since their *why* (one Planner, RBAC at the data layer, stateless agents, evidence-first) is still correct. |
| 12 | Redesigned the live `PlannerTask`/`AgentResult` to add `role` (`PlannerTask`) and `evidence`/`confidence` (`AgentResult`), and added a merged `evidence` list to `PlannerResponse` | This closes the single most concrete deviation the review found: CLAUDE.md calls `evidence` "non-negotiable... not decoration" (FR-013), but the live schema had no such field -- only the now-deleted dead scaffold did. `role` lives inside `PlannerTask` rather than as a second `execute(task, role)` parameter, so `BaseAgent`'s method signature never has to change again to add auth later. Populated `evidence`/`confidence` meaningfully in every real agent (not left as an always-empty placeholder) so the field is real, not cosmetic. |
| 13 | Introduced a per-domain Data Access Layer gateway (`SalesDataAccess`, `FinanceDataAccess`) between each agent and its existing `DataLoader` | Agents were constructing `SalesDataLoader`/`FinanceDataLoader` and calling `.load()` directly -- i.e. reading CSVs themselves, the exact thing CLAUDE.md's MCP/RBAC design says agents must never do. The gateway accepts (and, until auth exists, ignores) a `role` argument, so RBAC can be enforced inside this one method per domain later without touching the agent, `AnalyticsService`, or `QueryMapper`. `DataLoader` classes are unchanged and still do the actual CSV/pandas work underneath the gateway. |
| 14 | Added `app/planner/bootstrap.py:build_planner()` as the single place that registers agents into a `PlannerService` | `demo.py` and `app/api/deps.py` were independently constructing the same `AgentRegistry` with the same four agents -- a fifth agent meant remembering to edit both. Both callers now delegate to one function; `demo.py` keeps only its CLI-specific dataset-fallback-to-fixtures logic on top of it. |
| 15 | Refactored `EnterpriseAgent`/`EnterpriseAnalyticsService` to aggregate whatever domains are actually present in `task.metadata["upstream_results"]`, instead of hardcoding `upstream.get("sales")`/`upstream.get("finance")` | The review flagged that adding HR/Marketing/Inventory/Operations would require editing Enterprise-layer internals -- the same "special-case an agent" anti-pattern CLAUDE.md forbids for the Planner, recurring one layer up. `DOMAIN_FIELD_LABELS` is now a purely cosmetic, optional rename table (e.g. `sales.total_revenue` -> `sales_revenue`); a domain absent from it still aggregates correctly via a namespaced fallback (`hr_headcount`, etc.), proven by a test that registers a fake "hr" upstream result with zero `EnterpriseAgent` code changes. |
| 16 | Documented Postgres/Redis/ChromaDB/Scheduler/Sentinel/Alert Engine/full RBAC as explicitly deferred, with the exact seam each will attach to, rather than either building them now or leaving stale, disconnected stub folders | Direct project instruction: back this refactor with clean seams, not premature implementations. Seams are documented in `CLAUDE.md`'s "Deferred, seam-only" section: `PlannerResponse.task_id` as the future audit-log join key, a proactive caller building a `PlannerTask(source="sentinel")` through the same `build_planner()`, and `PlannerTask.role` plus each domain's Data Access Layer gateway (decision #13) as where RBAC attaches. |
| 17 | Consolidated `SalesQueryMapper`/`FinanceQueryMapper`/`EnterpriseQueryMapper`'s shared "first matching keyword rule wins, else default" mechanism into `app/common/keyword_mapper.py:KeywordMapper`, and the duplicated "does this text contain any of these keywords" check (in `router.py` and `demo.py`) into `matches_any()` | Three copies of the same small algorithm, each with its own rules table, is exactly the "quadruplicated keyword logic, easy to update three of four places" risk the review flagged (it had already happened once). Each domain's *rules* (genuinely different vocabulary) stay in that domain's own `query_mapper.py`; only the matching *mechanism* moved. |
| 18 | Removed `demo_sales.py`/`demo_finance.py` and trimmed `backend/requirements.txt` to only packages actually imported (`fastapi`, `uvicorn`, `pydantic`, `pandas`, `httpx`) | The two single-domain demos were fully superseded by the unified `demo.py` from an earlier feature and just sat as stale duplicates. `requirements.txt` previously listed `sqlalchemy`, `psycopg2-binary`, `redis`, `chromadb`, `langchain`, `langgraph`, `anthropic` -- none imported anywhere -- which misrepresented what's actually built. Re-add each one (with a Decision Log entry) exactly when its corresponding deferred seam (#16) is actually implemented. |
| 19 | Left `frontend/components/EvidenceTrail.tsx` unchanged rather than deleting it as "stale" | Its `EvidenceItem { source: string; data_point: string }` shape turned out to already exactly match the new live `EvidenceItem` model from decision #12 -- a side effect of restoring the original spec's evidence shape. It's unused only in the ordinary sense that no frontend page exists yet to render it, not because it's wrong. |

## Open questions / not yet decided
- Exact anomaly-detection method for the Enterprise Intelligence Engine (simple percentage-threshold vs. something more statistical) — currently planned as simple % change over a rolling window for MVP; can be revisited if time allows.
- Whether Marketing/Support/Operations agents get real implementations or remain stubs for the workshop demo — currently planned as stubs (see `02_PRD.md` MVP scope).
- Long-term data strategy for real CRM/ERP integration — deferred to Future Scope (`09_Development_Roadmap.md`).

## How to use this log going forward
Any time a new significant decision is made (e.g. changing the anomaly-detection approach, adding a new agent, changing the RBAC model), add a row here with the decision and the reasoning — not just an update to the architecture diagram. This is what lets any teammate, or a new Claude session, understand *why* the system looks the way it does, not just what it looks like.
